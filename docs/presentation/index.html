Factory Method Christian

Problem: 4 Fraktionen × 3 Einheiten = 12 Klassen — wie einheitlich erzeugen?

[Christian] Jede Fraktion hat eigene Factory. Template-Method-Ansatz in createUnit().

Factory Method — Code Christian

// UnitFactory.java (Abstract Creator)
public abstract class UnitFactory {
    public final Unit createUnit(UnitType type) {
        if (!isValidTypeForFaction(type))
            throw new IllegalArgumentException(/*...*/);
        return createUnitInternal(generateUnitId(), type);
    }
    protected abstract Unit createUnitInternal(String id, UnitType type);
}
// FireUnitFactory.java (Concrete Creator)
protected Unit createUnitInternal(String id, UnitType type) {
    return switch (type) {
        case INFERNO_WARRIOR -> new InfernoWarrior(id, new UnitStats(100, 15, 5, 3, 1));
        case FLAME_ARCHER   -> new FlameArcher(id, new UnitStats(70, 12, 3, 4, 3));
        case PHOENIX         -> new Phoenix(id, new UnitStats(80, 10, 4, 5, 1));
        default -> throw new IllegalArgumentException("Unknown type");
    };
}
[Christian] createUnit() als Template Method. Switch-Expression in Java 21.

Builder Max

Problem: Komplexe Spielkonfiguration mit vielen optionalen Parametern

[Max] GameBuilder ermöglicht schrittweisen Aufbau der Spielkonfiguration.

Builder — Code Max

// GameBuilder.java — Fluent API
public class GameBuilder {
    private final Set<Faction> factions = new LinkedHashSet<>();
    private Map<Terrain, Integer> customTerrainDistribution = null;

    public GameBuilder withFactions(Faction... factions) {
        this.factions.addAll(Arrays.asList(factions));
        return this;
    }

    public GameBuilder withCustomTerrain(Map<Terrain, Integer> dist) {
        this.customTerrainDistribution = new EnumMap<>(dist);
        return this;
    }

    public Game build() {
        validate();
        Battlefield battlefield = createAndInitializeBattlefield();
        Game game = new Game(battlefield);
        placeUnitsOnBattlefield(game);
        return game;
    }
}
[Max] Fluent API mit Method Chaining. build() validiert und erzeugt das Game-Objekt.

Composite Max

Problem: Einheitliche Operationen auf Zellen, Regionen und dem gesamten Battlefield

[Max] Cell → Region → Battlefield: gleiche Schnittstelle, applyEffect() propagiert.

Composite — Code Max

// BattlefieldComponent.java
public interface BattlefieldComponent {
    List<Cell> cells();

    default void applyEffect(Consumer<Cell> effect) {
        cells().forEach(effect);
    }
}
// Cell.java (Leaf)
public class Cell implements BattlefieldComponent {
    @Override
    public List<Cell> cells() { return List.of(this); }
}

// Region.java (Composite)
public record Region(List<Cell> cells) implements BattlefieldComponent {
    @Override
    public List<Cell> cells() { return new ArrayList<>(cells); }
}
[Max] Default-Methode applyEffect() iteriert über cells(). Leaf und Composite implementieren cells() unterschiedlich.

Decorator Christian

Problem: Dynamische Buffs/Debuffs ohne Klassenexplosion

[Christian] Decorators werden zur Laufzeit hinzugefügt und können ablaufen (isExpired).

Decorator — Code Christian

// UnitDecorator.java
public abstract class UnitDecorator {
    public abstract int getAttackBonus(Unit unit);
    public abstract int getDefenseBonus(Unit unit);
    public abstract boolean isExpired();
    public abstract void tick();
}
// Unit.java — Aggregation der Decorators
public int getAttack() {
    int attack = baseStats.attack();
    for (UnitDecorator decorator : decorators) {
        if (!decorator.isExpired()) {
            attack += decorator.getAttackBonus(this);
        }
    }
    return attack;
}
[Christian] Unit aggregiert Decorators. getAttack() summiert alle aktiven Boni. tick() zählt Runden herunter.

Strategy Max

Problem: Unterschiedliche Bewegungs- und Angriffslogik pro Einheitentyp

[Max] Ground vs. Flying Movement, Melee vs. Ranged Attack. Geländeabhängige Kosten.

Strategy — Code Max

// MovementStrategy.java
public interface MovementStrategy {
    double calculateMovementCost(Terrain terrain);
    boolean canMoveTo(Game game, Position current,
                      Position target, int maxMovement);
}
// FlyingMovementStrategy.java
public class FlyingMovementStrategy implements MovementStrategy {
    private static final double FLYING_TERRAIN_COST = 1.0;

    @Override
    public double calculateMovementCost(Terrain terrain) {
        return FLYING_TERRAIN_COST; // Gelände ignorieren
    }

    @Override
    public boolean canMoveTo(Game game, Position current,
                             Position target, int maxMovement) {
        if (game.isPositionOccupied(target)) return false;
        return current.manhattanDistanceTo(target) <= maxMovement;
    }
}
[Max] Fliegende Einheiten ignorieren Geländekosten. Bodeneinheiten haben variable Kosten.

Visitor Christian

Problem: 5 Gelände × 4 Fraktionen = 20 Effekte — ohne if-Kaskade

// TerrainVisitor.java
public interface TerrainVisitor {
    TerrainEffectResult visitFireUnit(Unit unit);
    TerrainEffectResult visitWaterUnit(Unit unit);
    TerrainEffectResult visitEarthUnit(Unit unit);
    TerrainEffectResult visitAirUnit(Unit unit);
}
// Unit.java — accept()
public TerrainEffectResult accept(TerrainVisitor visitor) {
    return switch (faction) {
        case FIRE  -> visitor.visitFireUnit(this);
        case WATER -> visitor.visitWaterUnit(this);
        case EARTH -> visitor.visitEarthUnit(this);
        case AIR   -> visitor.visitAirUnit(this);
    };
}
[Christian] Jeder Geländetyp hat eigenen Visitor. Double Dispatch über accept().

Chain of Responsibility Christian

5-stufige Schadensberechnung: Base → Faction → Terrain → Synergy → Defense

// DamageHandler.java
public abstract class DamageHandler {
    private DamageHandler next;

    public DamageHandler setNext(DamageHandler next) {
        this.next = next;
        return next;
    }

    public void handle(DamageContext context) {
        if (next != null) next.handle(context);
    }
}
// BaseDamageHandler.java
public class BaseDamageHandler extends DamageHandler {
    @Override
    public void handle(DamageContext context) {
        int baseDamage = context.getAttacker().getAttackStrategy()
            .calculateBaseDamage(context.getAttacker(), context.getTarget());
        context.setBaseDamage(baseDamage);
        super.handle(context); // weiter in der Kette
    }
}
[Christian] Jeder Handler modifiziert den DamageContext und reicht ihn weiter. Leicht erweiterbar.

Pattern-Zusammenspiel Max

Erzeugung: Builder erstellt Game mit Composite-Battlefield & Einheiten via Factory Method
Spielablauf: State steuert Phasen → Command kapselt Züge → Strategy bestimmt Bewegung/Angriff
Kampf: Chain of Resp. berechnet Schaden → nutzt Visitor (Gelände) & Decorator (Buffs)
Events: Observer benachrichtigt UI über alle Änderungen → State & Decorator reagieren
[Max] Patterns arbeiten zusammen: Builder baut das Spiel, State/Command steuern den Ablauf, Chain/Visitor/Decorator berechnen Kampf, Observer koppelt die UI.
